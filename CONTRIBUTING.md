# Contributing

感谢你关注 GdeiAssistant 的改进与维护。

## 基本流程

1. 先阅读 README、相关协议/政策页面和本文件，确认变更范围。
2. 从最新的 `main` / `master` 分支拉出功能分支。
3. 在本地完成开发、测试和必要的文档更新。
4. 提交 PR，说明改动目的、影响范围、测试结果和风险点。

## 分支与 PR 建议

- 分支名建议体现改动主题，例如 `feat/privacy-policy-update`、`fix/login-token-handling`。
- PR 标题应直接说明改动内容，避免使用过于模糊的描述。
- 涉及前端协议页面、隐私、登录、校园系统、对象存储、短信、邮件、AI OCR、爬虫/模拟登录等能力的 PR，请额外说明数据流、凭证流向、存储位置和风险控制措施。

## 代码风格

- 保持与现有代码风格、目录结构和命名习惯一致。
- 不要在没有必要的情况下引入新的框架、重型依赖或复杂抽象。
- 变更协议/政策或社区规则时，请避免凭空添加法律主体、备案号、许可证号、地址、负责人姓名、邮箱或其他无法从仓库明确证明的信息。

## 测试与构建

- 前端改动至少应保证 `frontend` 能通过构建，或不会引入明显语法错误。
- 后端改动应尽量补充或更新相关测试。
- 提交前建议运行与变更直接相关的测试或构建命令，并在 PR 中说明结果。

## 提交信息建议

建议使用清晰、可追踪的提交信息，例如：

- `feat: add third-party services policy page`
- `fix: align cookie notice with token storage implementation`
- `docs: add repository governance documents`

## 许可证与贡献授权

- 除非另有明确说明，向本仓库提交的代码、文档和其他贡献，默认按仓库当前采用的 Apache-2.0 许可证进行授权。
- 提交贡献前，请确认你有权提交相应内容，且不会侵犯第三方权利。

## 禁止提交的内容

请不要提交以下内容：

- 真实用户数据、真实学校账号、真实密码、真实验证码
- API Key、短信密钥、邮件密钥、AI API Key、对象存储密钥、数据库连接串、生产环境配置
- 未脱敏日志、抓包、截图、取件码、完整手机号、完整邮箱、完整证件号

## 文档同步

当改动涉及用户权利义务、个人信息处理、第三方服务、交易/跑腿/失物规则、安全说明或开源治理边界时，请同步评估是否需要更新：

- `README.md`
- 前端 About/政策页面
- `SECURITY.md`
- `THIRD_PARTY_NOTICES.md`
- `TRADEMARK.md`
- `NOTICE`
