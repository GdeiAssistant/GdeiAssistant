# Security Policy

## 漏洞报告方式

如果你发现可能影响 GdeiAssistant 的安全问题，请优先通过项目公开反馈渠道、Issue 或后续公布的安全联系方式提交。

- 对可直接利用、可能导致越权、账号接管、敏感信息泄露、远程执行或批量影响用户的漏洞，请避免在公开 Issue 中直接披露完整利用细节。
- 如需先公开占位，请仅描述影响范围和大致现象，并等待维护者引导到更适合补充细节的渠道。

## 支持版本

安全修复通常优先面向以下范围：

- 当前 `main` / `master` 分支
- 当前仍在维护的正式发布版本

历史分支或长期无人维护的私有部署不保证获得回溯修复。

## 安全测试边界

请勿在未经明确授权的情况下进行以下行为：

- 攻击学校相关系统、第三方系统或与本项目集成的外部平台
- 获取、尝试获取或导出他人数据
- 爆破账号、撞库、批量请求、绕过验证码、批量爬取、拒绝服务或其他破坏性测试
- 提交真实密码、真实验证码、真实 Token、真实取件码、真实证件号、真实手机号或其他高敏数据
- 对生产或公共演示环境执行会影响可用性、数据完整性或用户体验的测试

## 报告内容建议

请尽量提供以下信息：

- 漏洞描述
- 影响范围
- 复现步骤
- 截图、日志或抓包摘要
- 触发条件与利用前提
- 修复建议或缓解思路（如有）

## 响应说明

- 维护者通常会在合理期限内评估报告有效性、影响等级和修复优先级。
- 是否公开修复细节、何时发布补丁、是否回溯旧版本，由维护者结合风险和维护成本判断。
- 本项目当前不承诺奖金、赏金计划或固定 SLA。

## 敏感信息处理

请不要在公开 Issue、PR、评论或讨论中粘贴以下内容：

- 真实账号、密码、验证码、Token、Cookie、Session
- 完整手机号、完整邮箱、完整校园账号、完整证件号
- 完整取件码、详细住址、门禁信息、支付信息
- 未脱敏数据库记录、日志原文、环境变量或密钥
