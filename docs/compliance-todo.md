## 11. 充值请求客户端 HMAC 迁移

- 当前状态：已完成后端、Android、iOS 收口。后端不再将客户端提交的 `hmac` / `timestamp` 作为可信安全边界；Android 和 iOS 已移除客户端内置 HMAC 共享密钥；`REQUEST_VALIDATE_TOKEN` 已不再作为部署配置项使用。旧 `hmac` / `timestamp` 字段仅作为兼容输入保留并被忽略，充值仍依赖登录态、设备标识、二次密码验证、金额校验、限流和审计日志。
- 风险说明：若旧密钥曾公开或进入发布包，应视为已泄露；当前后端已不再依赖该密钥作为安全边界。
- 建议方案：兼容期后再移除旧 `hmac` / `timestamp` 字段；继续补全客户端幂等键复用、订单状态表和 UNKNOWN 补偿能力。
- 优先级：中
- 是否需要后端：是
- 是否需要运营人工处理：否

## 12. 充值请求 Redis 幂等键基础机制

- 当前状态：已实现。后端 `POST /api/card/charge` 已支持可选 `Idempotency-Key` / `X-Idempotency-Key`，并使用 Redis 原子写入短期幂等状态，避免同一用户、同一接口、同一幂等键、同一 payload fingerprint 的重复请求再次触发学校/支付链路；Android 已开始提交幂等键，iOS 后续接入后也可复用该机制。未提交幂等键的旧客户端仍保持兼容。
- 风险说明：Redis 只承担短期并发与重试合并，不能替代长期订单状态或学校侧到账确认。
- 建议方案：兼容期后再评估是否强制新客户端提交幂等键，并继续补全充值查询和 UNKNOWN 补偿能力。
- 优先级：高
- 是否需要后端：是
- 是否需要运营人工处理：视 UNKNOWN 补偿方案而定
